# Expo Native Modules & Config Plugins

The Expo Modules API provides a Swift/Kotlin/TypeScript DSL for writing native modules. `create-expo-module` scaffolds the boilerplate. Config plugins customize the native project during `expo prebuild`.

## Scaffolding with create-expo-module

**Local module** (lives inside your app, not published to npm):

```bash
npx create-expo-module@latest key-value-store --local --platform apple android
```

Creates `modules/key-value-store/` with Swift, Kotlin, and TypeScript files pre-wired.

**Standalone module** (publishable npm package):

```bash
npx create-expo-module@latest expo-key-value-store --platform apple android
```

Generates a package with an example app.

**Platform flags:** `apple`, `android`, `web`. Default is all three.

**Feature examples** (generates working code patterns):

```bash
npx create-expo-module@latest my-module --local \
  --platform apple android \
  --feature Function AsyncFunction Event View
```

Available features: `Function`, `AsyncFunction`, `Event`, `View`, `SharedObject`, `Constant`.

**Add platform support to an existing module:**

```bash
npx create-expo-module@latest add-platform-support --platform android
```

## Module Definition DSL

All platforms use the same DSL pattern:

**iOS (Swift) — `ios/MyModule.swift`:**

```swift
import ExpoModulesCore

public class MyModule: Module {
  public func definition() -> ModuleDefinition {
    Name("MyModule")

    Constants([
      "PI": Double.pi
    ])

    Function("multiply") { (a: Double, b: Double) -> Double in
      return a * b
    }

    AsyncFunction("fetchData") { (url: String, promise: Promise) in
      // Async work here
      promise.resolve(["result": "data"])
    }

    Events("onProgress", "onComplete")

    View(MyView.self) {
      Prop("color") { (view: MyView, color: UIColor) in
        view.backgroundColor = color
      }
      Events("onTap")
    }
  }
}
```

**Android (Kotlin) — `android/src/main/java/.../MyModule.kt`:**

```kotlin
import expo.modules.kotlin.modules.Module
import expo.modules.kotlin.modules.ModuleDefinition

class MyModule : Module() {
  override fun definition() = ModuleDefinition {
    Name("MyModule")

    Constants(
      "PI" to Math.PI
    )

    Function("multiply") { a: Double, b: Double ->
      a * b
    }

    AsyncFunction("fetchData") { url: String ->
      mapOf("result" to "data")
    }

    Events("onProgress", "onComplete")
  }
}
```

**TypeScript bindings — `src/MyModule.ts`:**

```ts
import { requireNativeModule } from 'expo-modules-core';

const MyModule = requireNativeModule('MyModule');

export function multiply(a: number, b: number): number {
  return MyModule.multiply(a, b);
}

export async function fetchData(url: string): Promise<{ result: string }> {
  return MyModule.fetchData(url);
}

export const PI: number = MyModule.PI;
```

## Module Registration — expo-module.config.json

```json
{
  "platforms": ["apple", "android"],
  "apple": {
    "modules": ["MyModule"]
  },
  "android": {
    "modules": ["com.mypackage.MyModule"]
  }
}
```

Note: iOS uses simple class names; Android requires the fully-qualified package path.

## Shared Objects

Shared Objects keep native instances alive across JS calls (useful for long-lived connections like Bluetooth, sensors):

```swift
// iOS
class BluetoothSession: SharedObject {
  var connection: CBCentralManager?
}

// Module definition
SharedObjectType(BluetoothSession.self) {
  Function("connect") { (session: BluetoothSession) in
    session.connection = CBCentralManager()
  }
}
```

## Config Plugins

Config plugins customize the native project generated by `expo prebuild`. They run synchronously and must return serializable values (except `mods`).

**Directory structure:**

```
my-module/
├── plugin/
│   ├── src/
│   │   └── index.ts    ← plugin source
│   └── build/
│       └── index.js    ← compiled output
└── app.plugin.js       ← entry point (points to plugin/build/index.js)
```

**iOS — modify Info.plist:**

```ts
import { ConfigPlugin, withInfoPlist } from '@expo/config-plugins';

const withCameraPermission: ConfigPlugin<{ message?: string }> = (
  config,
  { message = 'Allow camera access' } = {}
) => {
  return withInfoPlist(config, (config) => {
    config.modResults['NSCameraUsageDescription'] = message;
    return config;
  });
};

export default withCameraPermission;
```

**Android — modify AndroidManifest.xml:**

```ts
import { withAndroidManifest } from '@expo/config-plugins';

const withPermission: ConfigPlugin = (config) => {
  return withAndroidManifest(config, (config) => {
    const manifest = config.modResults.manifest;
    manifest['uses-permission'] = [
      ...(manifest['uses-permission'] ?? []),
      { $: { 'android:name': 'android.permission.CAMERA' } },
    ];
    return config;
  });
};
```

**Declare in app.json:**

```json
{
  "expo": {
    "plugins": [
      ["my-module", { "message": "We need camera for scanning" }]
    ]
  }
}
```

**Build and test:**

```bash
npm run build plugin          # compile TypeScript plugin
npx expo prebuild --clean     # test plugin application
```

## Key Points

- Use `--local` for app-specific modules; omit for publishable packages
- The DSL is identical across Swift and Kotlin — `Name`, `Function`, `AsyncFunction`, `Events`, `View`, `SharedObject`
- Config plugins must be synchronous; `mods` (async operations) run during `prebuild`
- Re-run `npx expo prebuild` after changing a config plugin; re-run `eas build` after changing native module code
